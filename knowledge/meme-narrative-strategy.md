# Meme 代币叙事总结与热门推文方案

## 问题

1. **叙事总结不够准确**：仅通过 OpenNews 按 symbol/name 搜索，对小众 meme 代币覆盖率低
2. **推特喊单抓取不精准**：仅用 symbol 作为关键词，匹配到大量无关推文
3. **API 成本高**：每次页面访问都调用第三方 API，浪费配额

## 方案架构

```
┌─────────────────────────────────────────────────┐
│                 代币详情页                         │
│  ┌──────────────┐  ┌──────────────────────────┐  │
│  │  叙事总结卡片  │  │  热门推特侧边栏            │  │
│  └──────┬───────┘  └────────────┬─────────────┘  │
│         │                       │                 │
│  GET /narrative              GET /tweets          │
└─────────┼───────────────────────┼─────────────────┘
          │                       │
    ┌─────▼───────────────────────▼─────┐
    │     Supabase 持久化缓存层          │
    │  token_narratives / token_tweets  │
    │  TTL: 叙事 4h / 推文 2h           │
    └─────┬───────────────────────┬─────┘
          │ cache miss            │ cache miss
    ┌─────▼─────┐          ┌─────▼──────┐
    │ OpenNews  │          │ OpenTwitter│
    │ 多策略搜索 │          │ 多维搜索    │
    └───────────┘          └────────────┘
```

## 五层优化策略

### 第一层：多策略搜索（提升准确度）

#### 叙事搜索策略（OpenNews）

| 策略 | 方法 | 说明 |
|------|------|------|
| coins 过滤 | `coins: ["PEPE"]` | 匹配已标注代币的新闻，最精准 |
| 名称关键词 | `q: "Pepe"` | 覆盖标题/正文提及的新闻 |
| meme 引擎 | `engineTypes: { meme: [] }` + `q` | 专门的 meme 新闻引擎，覆盖小众代币 |
| cashtag | `q: "$PEPE"` | 加密社区常用格式，匹配推特式标记 |

结果按 AI 评分排序，去重后取 Top 8 生成摘要。

#### 推特搜索策略（OpenTwitter）

| 策略 | 方法 | 说明 |
|------|------|------|
| cashtag | `keywords: "$PEPE"` | 加密推特最标准的搜索方式 |
| 合约地址前缀 | `keywords: "So1111..."` (前 10 位) | 抓取分享合约地址的推文 |
| 代币名称 | `keywords: "Pepe"` (≥4 字符) | 补充搜索，仅名称够独特时使用 |

### 第二层：Supabase 持久化缓存（降低成本）

- `token_narratives` 表：存储叙事总结，TTL 4 小时
- `token_tweets` 表：存储推文列表，TTL 2 小时
- 应用重启不丢失缓存，同一代币 4 小时内只调用一次 API
- **成本节省**：假设 20 个代币 × 每天 100 次页面访问 = 2000 次 API 调用 → 降至 ≈120 次

### 第三层：批量预取（主动刷新）

- 榜单每次更新后，后台异步预取所有代币的叙事
- 并发控制：每次 2 个代币，批次间隔 3 秒
- 推文预取默认关闭（`ENABLE_TWEET_PREFETCH=false`），需手动开启
- 用户访问详情页时直接命中缓存，零延迟

### 第四层：智能过滤（提升精准度）

#### 相关性检查
推文必须实际提及目标代币才会展示：
- 包含 cashtag（`$PEPE`）
- 包含合约地址前 8 位
- 包含代币名称（≥4 字符）
- 短 symbol（≤3 字符）必须是独立词或 cashtag

#### 营销号过滤（增强版）
- 文案关键词检测：30+ 条营销话术
- 账号 bio 检测：20+ 条营销标记
- 用户名模式匹配：bot/shill/signal/pump 等
- 粉丝质量检查：粉丝数 < 100 或关注/粉丝比 < 0.1
- emoji 密度检查：emoji 超过文本 15% 的过滤

### 第五层：KOL 加权排序（提升质量）

推文按综合质量分排序，而非仅看互动量：

```
质量分 = 互动分 × KOL 系数 × 新鲜度系数

互动分 = 点赞×1.0 + 转发×2.0 + 回复×0.5

KOL 系数:
  已认证账号: +0.5
  粉丝 ≥100K: +1.0
  粉丝 ≥50K:  +0.7
  粉丝 ≥10K:  +0.4
  粉丝 ≥1K:   +0.1

新鲜度系数:
  24h 内: ×1.2
  48h 内: ×1.0
  超过 48h: ×0.8
```

## 配置说明

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `OPENNEWS_TOKEN` | - | 6551 API Token（叙事必需） |
| `TWITTER_TOKEN` | - | 6551 API Token（推文必需，与 OPENNEWS_TOKEN 互通） |
| `NARRATIVE_CACHE_TTL_HOURS` | 4 | 叙事缓存有效期（小时） |
| `TWEET_CACHE_TTL_HOURS` | 2 | 推文缓存有效期（小时） |
| `ENABLE_TWEET_PREFETCH` | false | 榜单更新后是否自动预取推文 |

## 数据库

执行 `config/sql/token-narrative-cache.sql` 创建缓存表。
不创建也可运行，系统会降级为仅内存缓存（重启丢失）。

## 情绪分析

叙事总结自动分析新闻情绪：

| sentiment | 条件 |
|-----------|------|
| `bullish` | 看多信号 > 看空信号 × 2 |
| `slightly_bullish` | 看多 > 看空 |
| `neutral` | 看多 ≈ 看空 |
| `slightly_bearish` | 看空 > 看多 |
| `bearish` | 看空信号 > 看多信号 × 2 |

## 未来优化方向

1. **LLM 摘要**：用 AI 对多条新闻/推文生成结构化叙事，而非简单拼接
2. **KOL 列表**：维护 crypto KOL 白名单，搜索他们的推文作为叙事补充
3. **合约地址关联**：解析推文中的链接/合约地址，精确关联代币
4. **WebSocket 实时更新**：订阅 OpenNews 实时新闻流，自动更新叙事
5. **多语言叙事**：同时保留中英文摘要，前端按用户语言展示
