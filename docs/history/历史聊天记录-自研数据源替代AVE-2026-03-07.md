# 历史聊天记录：自研数据源替代 AVE Cloud API（2026-03-07）

本记录保存「实现自研数据源完全替代 AVE Cloud API」的需求、方案设计、实现与调试全过程。

## 1. 需求背景

使用 AVE Cloud API 存在以下问题：

- **请求限制**：免费版仅 1 RPS（每秒 1 次请求），拉取 200+ 代币需要数分钟
- **收费问题**：高级功能（WebSocket 实时流、更高速率）需付费
- **依赖风险**：核心数据流完全依赖单一第三方 API，一旦停服或变更接口将导致榜单功能不可用

**目标**：实现功能对等的自研数据源模块，无需 API Key、无额外费用，替代项目中所有 AVE 调用。

## 2. 方案设计

### 2.1 数据源选型

经调研，选择以下 4 个免费 API 聚合替代 AVE：

| 数据源 | 用途 | 速率限制 | 需要 Key |
|--------|------|----------|----------|
| **DexScreener** | 代币搜索、交易对详情、boost/profiles 列表 | 300 req/min | 否 |
| **GeckoTerminal** | 趋势池、新池、OHLCV K 线、交易记录 | ~20 req/min | 否 |
| **Jupiter** | Solana 代币价格（v2 现需认证，作为可选补充） | — | 是（v2） |
| **GoPlus Security** | 合约安全检测（蜜罐、可增发、可冻结等） | 30 req/min | 否 |
| **Binance Web3** | Top10 持有人占比、持币地址数、insider 占比 | 宽松 | 否 |

### 2.2 AVE 功能映射

| AVE 功能 | 替代方案 |
|----------|----------|
| 代币搜索 `/v2/tokens?keyword=` | DexScreener search + GeckoTerminal searchPools |
| 平台代币 `/v2/tokens/platform?tag=pump_in_hot` | DexScreener 多关键词搜索 + boosts/profiles + GeckoTerminal trending |
| 平台代币 `/v2/tokens/platform?tag=pump_in_new` | DexScreener 多关键词搜索 + profiles + GeckoTerminal new_pools |
| 代币详情 `/v2/tokens/{addr}-{chain}` | DexScreener token pairs + GeckoTerminal token info |
| 排行榜 `/v2/ranks?topic=` | DexScreener search + GeckoTerminal trending |
| K 线 `/v2/klines/` | GeckoTerminal OHLCV |
| 交易记录 `/v2/txs/` | GeckoTerminal pool trades |
| 合约风险 `/v2/contracts/` | GoPlus Security API（Solana 专用端点） |
| 批量价格 `/v2/tokens/price` | Jupiter + DexScreener |
| LP 锁定 / insider 检查 | GoPlus + Binance Web3 |

### 2.3 架构设计

```
src/data-sources/
├── rate-limiter.js      # 令牌桶限流器
├── chain-map.js         # 跨平台链 ID 映射
├── dexscreener.js       # DexScreener API 客户端
├── geckoterminal.js     # GeckoTerminal API 客户端
├── jupiter.js           # Jupiter Price API 客户端
├── goplus.js            # GoPlus Security API 客户端（EVM + Solana 双端点）
└── index.js             # 统一聚合层，AVE 兼容格式输出
```

统一聚合层 `index.js` 对外暴露与 AVE 等价的函数接口（`getPlatformTokens`、`getTokenDetail`、`getRanks` 等），内部自动选择最佳数据源、处理降级与缓存。

## 3. 实现过程

### 3.1 基础模块

- **rate-limiter.js**：基于令牌桶的限流器，支持并发排队，确保各 API 不超速
- **chain-map.js**：统一映射 `solana / eth / bsc / base / arbitrum / polygon / avalanche / optimism / ton` 在 DexScreener、GeckoTerminal、GoPlus 之间的链标识

### 3.2 API 客户端

- **dexscreener.js**：搜索、批量获取代币交易对、boosts/profiles 列表；`normalizePair()` 将 DexScreener pair 转为 AVE 兼容格式
- **geckoterminal.js**：趋势池/新池分页获取、代币信息、OHLCV K 线、交易记录、网络列表；`normalizePool()` 解析 JSON:API 格式的 included/relationships
- **jupiter.js**：Solana 代币批量价格查询（v2 现需认证，自动 fallback 到空结果）
- **goplus.js**：区分 Solana 专用端点 `/api/v1/solana/token_security` 和 EVM 通用端点；Solana 检测字段包括 mintable/freezable/closable/metadata_mutable

### 3.3 统一聚合层（index.js）

- 3 分钟内存缓存，避免同一更新周期重复请求
- `getPlatformTokens(tag, limit)`：DexScreener 为主力（7 个 hot 关键词 + 6 个 new 关键词多源搜索 + boosts + profiles），GeckoTerminal 为补充
- `getTokenDetail(addr, chain)`：DexScreener + GeckoTerminal 双源取详情，Jupiter 补充价格
- `batchGetTokenSecurity(chain, addrs)`：GoPlus 批量安全检测
- `getTokenPrices(tokens)`：Jupiter（Solana）+ DexScreener（其他链）

### 3.4 脚本改造

- **fetch-pump-ranking.js**：移除所有 AVE 调用；并行拉取 hot/new/ranked；GoPlus 批量安全检测替代逐个 AVE token 详情；Binance 获取 holders + insiderPercent + top10Percent
- **fetch-zhilabs-ranking.js**：用 `dataSource.getTokenDetail()` 替代 AVE token 详情；增加 GoPlus + Binance 双重 fallback 获取持币地址数
- **check-token-eligibility.js**：全面改用自研数据源 + GoPlus 安全检测
- **server.js**：移除 AVE_API_KEY 必需提示

### 3.5 新增脚本

- **scripts/token-audit.js**：GoPlus 驱动的代币合约安全审计（替代原 ave-token-audit.js）
- **scripts/test-data-source.js**：数据源集成测试（20 项检查）

## 4. 调试与问题修复

### 4.1 GeckoTerminal chain 字段为空

**问题**：`normalizePool()` 从 `relationships.network.data.id` 提取链标识，但 GeckoTerminal 实际响应中无 `network` 字段。

**修复**：从 pool.id 前缀提取（`"solana_xxx"` → `"solana"`）。

### 4.2 Jupiter v2 需要认证

**问题**：`https://api.jup.ag/price/v2` 返回 401 Unauthorized。

**处理**：Jupiter 作为可选补充，价格数据主要由 DexScreener + GeckoTerminal 提供。

### 4.3 GoPlus Solana 端点不同

**问题**：通用端点 `/api/v1/token_security/solana` 返回空结果。

**修复**：Solana 使用专用端点 `/api/v1/solana/token_security`，响应格式也不同（mintable/freezable/closable 替代 is_honeypot/buy_tax/sell_tax）。

### 4.4 持币地址数显示 "—"

**问题**：GoPlus 仅对部分代币返回 `holder_count`（新/小代币无数据）。

**修复**：增加 Binance Web3 API 作为 fallback，从 `json.data.holders` 获取持币地址数（覆盖率接近 100%）。

### 4.5 Pump 榜单仅 6 条（目标 20 条）

**问题**：GeckoTerminal 频繁 429 限流（实际限制比文档更严格），导致候选池严重不足。

**根因分析**：
- 串行调用 `getPlatformTokens('pump_in_new')` → `getPlatformTokens('pump_in_hot')` → `getRanks('solana')`
- GeckoTerminal 20 req/min 在第 2 次调用时已耗尽
- 候选池从应有的 200+ 缩小到 12 条

**修复**：
1. DexScreener 作为主力数据源（300 req/min），7+6 个关键词多源并行搜索
2. DexScreener boosts + profiles 列表补充候选
3. `fetchAllTokens()` 改为 `Promise.all()` 并行拉取
4. 缓存 TTL 从 1 分钟增加到 3 分钟
5. `getRanks()` 增加 DexScreener fallback

**效果**：候选池从 12 条恢复到 229 条，筛选后从 6 条提升到 19 条。

## 5. 最终数据流

### Pump 榜单更新流程

```

+ GeckoTerminal trending + new (补充)
→ 229 条去重 Solana 代币

```




## 8. 已知限制

1. **Pump 榜单条数取决于市场状态**：筛选条件（市值>100K、上线<10天、Top10≤30%）较严格，候选池大小随市场波动。通常 15-20 条，极端情况可能少于 15 条。
2. **GeckoTerminal 限流**：免费 API 实际限制比文档更严格（~20 req/min），频繁调用会 429。代码已用 DexScreener 作为主力、GeckoTerminal 作为补充来缓解。
3. **Jupiter v2 需要认证**：价格数据由 DexScreener + GeckoTerminal 完整覆盖，Jupiter 仅作可选补充。
4. **GoPlus Solana 覆盖率有限**：新/小代币可能无 GoPlus 安全数据，持币地址数依赖 Binance Web3 fallback。
5. **无 insider_wallet_rate**：AVE 的 insider 指标无免费替代，改用 GoPlus mintable/freezable + Binance insiderHoldingPercent 作为风控补充。
