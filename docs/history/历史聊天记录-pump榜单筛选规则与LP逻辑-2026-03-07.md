# 历史聊天记录：Pump 榜单筛选规则与 LP 逻辑（2026-03-07）

本记录保存 Pump 榜单「Top10 占比」「LP 是否 burn」筛选规则的设计与修改过程，以及 0 条、本地/线上差异等排错结论。

## 1. 需求与实现概要

- **Top10 持有人占比**：超过 30% 的代币不入榜；数据来自 Binance Web3 token dynamic API（`top10HoldersPercentage`），表增加 `holders_top10_percent`，前端增加「Top10%」列。
- **LP 未 burn 不入榜**：仅当 LP 已 burn 或已锁定的代币可入榜。实现上依赖 AVE token 详情的 `is_lp_not_locked`；表增加 `lp_burned`，前端增加「LP」列（已burn/锁 / — / 否）。

## 2. LP 规则演进

- **最初**：只排除 `is_lp_not_locked === true`，且只对「LP 未知」的候选调 AVE token 详情补全；platform 返回 `false` 的未再校验，导致「实际未知」代币（如某 CA）仍入榜。
- **收紧**：对所有候选都调 AVE token 详情，仅保留「详情明确返回已 burn/锁定」；未知、未返回、接口失败一律排除 → 更新后**整表 0 条**（AVE 对多数 Solana pump 不返该字段）。
- **当前**：改为「仅排除明确未锁定」：详情返回 `true` 才排除；返回 `false` 或 **未知/null/失败** 均允许入榜，LP 列对未知显示「—」。常量 `EXCLUDE_LP_NOT_LOCKED_ONLY = true`，逻辑在 `scripts/fetch-pump-ranking.js`。

## 3. 0 条与「表未清空」问题

- 若筛选后 0 条，脚本原先直接 `return []` 未清空表，前端仍展示旧数据。已改为：**无论是否有新数据都先清空表**，再插入或 return，保证「更新完成: 0条」时前端显示「暂无数据」。

## 4. 本地与线上差异

- 用户在前端点击「更新 Pump 榜单」后终端出现「未知/未锁定/失败 均排除」「仅保留 LP 已 burn/锁定（详情明确）: 0 条」，与当前代码中的「仅排除明确未锁定」「排除 LP 明确未锁定 后候选」不一致。
- 说明：用户为**本地 `npm start`**，未部署线上。原因是 **Node 进程在启动时已加载脚本，改文件后未重启**，仍执行旧逻辑。处理：**重启本地服务**（Ctrl+C 后再次 `npm start`），再点「更新 Pump 榜单」即可使用新规则。

## 5. 单代币入榜校验

- 脚本 `scripts/check-token-eligibility.js`：对指定 token 拉取 AVE token 详情 + Binance Top10%，按当前规则逐条判断并输出能否入榜及原因。用法：`node scripts/check-token-eligibility.js <token_address>`。
